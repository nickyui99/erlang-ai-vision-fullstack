# Camera Playback & Download Implementation Plan

## Objective

Add a Tapo-style **Playback & Download** section to the existing camera control screen so users can review saved camera clips directly from the camera interface.

The first version should focus on event-based playback clips that already exist in the backend. The longer-term recording model should show 30-minute playback blocks in the UI, while storing video internally as smaller 5-minute recording segments. Events should be saved as timestamps that jump into the correct segment.

## Target Screen

Primary frontend file:

- `lib/features/dashboard/device_control_view.dart`

The playback section should appear inside `DeviceControlView`, near the live camera controls, similar to the reference screenshot:

- Live camera preview at the top
- Camera action bar below the preview
- Playback & Download card below the camera controls
- Recent clip thumbnails in a horizontal row

## MVP Scope

The MVP should reuse existing event clips instead of introducing a new recording timeline API.

Implement:

1. Load recent clips for the selected camera.
2. Show a **Playback & Download** panel in the camera screen.
3. Display recent clips as compact thumbnail-style cards.
4. Let the user tap a clip to request a signed playback URL.
5. Show a playback modal or open the playback URL.
6. Provide a download/open action using the same signed URL.

Defer:

- Full 24/7 recording timeline
- Calendar/date scrubber
- Cloud/local recording source switcher
- Native file download support on mobile
- Clip deletion and retention management
- Automatic stitching between adjacent 5-minute recording segments

## Clip Loading Flow

Add a method to `DeviceControlView`:

```dart
Future<void> _loadPlaybackClips() async {
  setState(() {
    _isLoadingPlayback = true;
    _playbackError = null;
  });

  try {
    final events = await widget.apiClient.listEvents();
    final deviceEvents = events
        .where((event) => event.deviceId == _device.deviceId)
        .take(6)
        .toList();

    final clips = <MediaClip>[];
    for (final event in deviceEvents) {
      final eventClips = await widget.apiClient.listEventClips(event.eventId);
      clips.addAll(
        eventClips.where(
          (clip) =>
              clip.deviceId == _device.deviceId &&
              clip.clipType == 'event' &&
              clip.status == 'available',
        ),
      );
    }

    if (!mounted) return;
    setState(() => _playbackClips = clips);
  } catch (error) {
    if (!mounted) return;
    setState(() => _playbackError = error.toString());
  } finally {
    if (mounted) setState(() => _isLoadingPlayback = false);
  }
}
```

Call `_loadPlaybackClips()` from:

- `initState()`
- `_refreshDevice()`
- after successful event refresh

## Playback URL Flow

Add a method:

```dart
Future<void> _playClip(MediaClip clip) async {
  setState(() {
    _playingClipId = clip.clipId;
    _playbackError = null;
  });

  try {
    final playback = await widget.apiClient.signedClipPlaybackUrl(clip.clipId);
    if (!mounted) return;
    setState(() => _selectedPlaybackUrl = playback);
    await _showPlaybackSheet(clip, playback);
  } catch (error) {
    if (!mounted) return;
    setState(() => _playbackError = error.toString());
    _toast(error.toString());
  } finally {
    if (mounted) setState(() => _playingClipId = null);
  }
}
```

## Phase: Recording Management Model

Use a two-layer recording model: short event clips for quick review, and continuous recording segments for timeline playback.

### Product Behavior

The camera screen should present playback in **30-minute blocks** because this is easy for users to scan and matches common camera-app behavior.

Example UI grouping:

```text
09:00 - 09:30
  09:07 Person detected
  09:18 Motion detected
  09:24 Unknown object detected

09:30 - 10:00
  09:42 Person detected
  09:55 Motion detected
```

When the user taps an event, playback should start slightly before the event timestamp, for example 10 seconds before detection.

```text
Event timestamp: 09:07:23
Playback start:  09:07:13
```

### Storage Behavior

Do not store physical video files as 30-minute or 1-hour files. Store continuous recordings as smaller **5-minute segments**.

Example storage layout:

```text
09:00 - 09:30 playback block
  09:00 - 09:05 segment
  09:05 - 09:10 segment
  09:10 - 09:15 segment
  09:15 - 09:20 segment
  09:20 - 09:25 segment
  09:25 - 09:30 segment
```

Why 5-minute segments:

- Easier to upload from edge devices.
- Smaller retry cost when uploads fail.
- Faster playback startup.
- Easier signed URL expiry management.
- Easier lifecycle deletion in OSS.
- Better seeking behavior on mobile.

### Event Clips vs Continuous Segments

Use both data types:

| Type | Length | Purpose | Table |
| --- | --- | --- | --- |
| Event clip | 30-60 seconds | Fast preview, alert review, download | `clips` |
| Recording segment | 5 minutes | Continuous timeline playback | `recordings` |
| UI block | 30 minutes | Playback navigation and grouping | computed by API/UI |

Event clips should remain tied to `event_id`. Recording segments should be tied to `device_id`, `start_time`, and `end_time`.

### Data Model Direction

The existing `recordings` table should represent physical 5-minute segments, not 30-minute UI blocks.

Recording segment fields:

```text
recording_id
device_id
user_id
start_time
end_time
storage_type
oss_object_key
duration_seconds
file_size_bytes
mime_type
status
retention_until
```

Event timestamp fields already exist through the event model:

```text
event_id
device_id
timestamp
event_type
summary
severity
```

For easier playback jumps, later add either:

```text
recording_id
recording_offset_seconds
```

or resolve the segment dynamically by querying:

```text
recording.start_time <= event.timestamp < recording.end_time
```

For the first implementation, dynamic lookup by timestamp is enough.

### Recording Query API

Add this API in the continuous playback phase:

```http
GET /api/v1/devices/{device_id}/recordings?from=2026-07-01T09:00:00Z&to=2026-07-01T09:30:00Z
```

Response should return all 5-minute segments inside the requested range, plus event markers in the same window.

Suggested response shape:

```json
{
  "data": {
    "device_id": "device_123",
    "from": "2026-07-01T09:00:00Z",
    "to": "2026-07-01T09:30:00Z",
    "segments": [
      {
        "recording_id": "rec_0900",
        "start_time": "2026-07-01T09:00:00Z",
        "end_time": "2026-07-01T09:05:00Z",
        "duration_seconds": 300,
        "status": "available"
      }
    ],
    "events": [
      {
        "event_id": "evt_123",
        "timestamp": "2026-07-01T09:07:23Z",
        "event_type": "person_detected",
        "summary": "Person detected near entrance"
      }
    ]
  }
}
```

Signed playback URLs for individual recording segments should be requested separately so URLs stay short lived.

### Playback Behavior

When a user taps a 30-minute block:

1. Frontend requests the block range.
2. Backend returns 5-minute recording segments and event markers.
3. Frontend displays a scrubber/timeline with event ticks.
4. Frontend requests a signed URL for the first needed segment.
5. When playback reaches the end of a segment, frontend requests the next segment URL.
6. If the user taps an event, frontend starts playback from the segment containing that timestamp.

### Recommended Timeline Defaults

- UI block size: 30 minutes.
- Physical recording segment size: 5 minutes.
- Event pre-roll: 10 seconds.
- Event post-roll for highlight clips: 20-45 seconds.
- Event clip length: 30-60 seconds.
- Recording retention: 7-30 days depending on storage cost.
- Failed incomplete recording uploads: expire after 1 day.

## Phase: Alibaba Cloud OSS Storage

This phase makes playback reliable by ensuring camera clips are uploaded to Alibaba Cloud OSS and played back through short-lived signed URLs.

### Goals

- Store event clips in Alibaba Cloud OSS.
- Keep only metadata and OSS object keys in the database.
- Generate short-lived upload URLs for edge devices.
- Generate short-lived playback URLs for authenticated app users.
- Prevent the Flutter app and edge device from using permanent cloud credentials.

### Backend Configuration

Confirm these backend settings exist and are documented:

```env
ALICLOUD_ACCESS_KEY_ID=
ALICLOUD_ACCESS_KEY_SECRET=
ALICLOUD_OSS_ENDPOINT=
ALICLOUD_OSS_BUCKET=
ALICLOUD_OSS_REGION=
ALICLOUD_OSS_UPLOAD_EXPIRES_SECONDS=900
ALICLOUD_OSS_PLAYBACK_EXPIRES_SECONDS=900
```

The backend should load these values through the existing config layer and expose no secrets to the frontend.

### OSS Object Key Design

Use predictable object keys that are easy to audit and lifecycle-manage:

```text
users/{user_id}/devices/{device_id}/events/{event_id}/clips/{clip_id}.mp4
users/{user_id}/devices/{device_id}/events/{event_id}/thumbnails/{clip_id}.jpg
```

Rules:

- Never trust an object key supplied directly by the frontend.
- Generate object keys on the backend.
- Bind each object key to `user_id`, `device_id`, `event_id`, and `clip_id`.
- Store the final key in `clips.oss_object_key`.

### Edge Upload Flow

The intended upload flow is:

1. Edge device detects an event and creates a local clip.
2. Edge calls the backend clip upload endpoint with `event_id`, `clip_type`, `mime_type`, `duration_seconds`, `file_size_bytes`, and `idempotency_key`.
3. Backend verifies the edge token and device ownership.
4. Backend creates or reuses a `Clip` row.
5. Backend generates an OSS signed upload URL.
6. Edge uploads the clip directly to OSS using the signed URL.
7. Edge calls the complete endpoint.
8. Backend marks the clip as `available`.

Expected clip status transitions:

```text
pending_upload -> available
pending_upload -> failed
available -> deleted
```

### Backend API Work

Verify or implement these edge-facing endpoints:

```http
POST /api/v1/edge/clips/upload-url
POST /api/v1/edge/clips/{clip_id}/complete
```

Verify or implement this app-facing endpoint:

```http
POST /api/v1/clips/{clip_id}/signed-url
```

The app-facing playback endpoint must:

- Require an authenticated user.
- Verify the clip belongs to the current user.
- Reject deleted clips.
- Reject clips that are not `available`.
- Reject clips with no `oss_object_key`.
- Return a short-lived signed playback URL.

### Frontend Integration

The Flutter app should continue using:

```dart
widget.apiClient.signedClipPlaybackUrl(clip.clipId)
```

No OSS credentials or bucket details should be added to Flutter.

The playback panel only needs:

- `clip.status`
- `clip.durationSeconds`
- `clip.uploadCompletedAt`
- signed playback URL returned by the backend

### OSS Lifecycle and Retention

Configure OSS lifecycle rules outside the app for cost control.

Recommended starting policy:

- Event clips: retain for 30 days.
- Thumbnails: retain for 30 days.
- Failed or incomplete uploads: expire after 1 day.

Keep application-level retention metadata in:

- `clips.expires_at`
- `recordings.retention_until`

### OSS Security Checklist

- Use least-privilege RAM credentials for the backend.
- Backend RAM policy should allow access only to the selected bucket and expected prefix.
- Signed upload URLs should be short lived.
- Signed playback URLs should be short lived.
- Do not make the OSS bucket public.
- Do not store signed URLs in the database.
- Log clip IDs and object keys, but never log access key secrets.
- Validate MIME type and file size before issuing upload URLs.
- Consider checksum validation during clip completion.

## Backend Follow-Up

The current frontend must load events first, then load clips per event. This is acceptable for the MVP but not ideal long term. Continuous playback should later use the `recordings` API so the frontend can request a 30-minute block and receive the underlying 5-minute segments plus event markers.

Add backend endpoints later:

```http
GET /api/v1/devices/{device_id}/clips?limit=20
GET /api/v1/devices/{device_id}/recordings?from=&to=
```

Suggested behavior:

- Require authenticated user
- Verify device belongs to current user
- Return available clips for that device
- Order by newest first
- Support `limit`
- Optional filters: `clip_type`, `status`, `from`, `to`

This would simplify event clip loading to one request:

```dart
widget.apiClient.listDeviceClips(_device.deviceId)
```

For continuous playback, the frontend should request a 30-minute range:

```dart
widget.apiClient.listDeviceRecordings(
  _device.deviceId,
  from: blockStart,
  to: blockEnd,
)
```

## Acceptance Criteria

The feature is complete when:

- The camera screen shows a **Playback & Download** panel.
- Recent available event clips for that camera appear in a horizontal row.
- Users can tap a clip and receive a playable signed URL.
- Users can open or download the clip.
- Available clips are backed by Alibaba Cloud OSS object keys.
- Continuous recording design uses 30-minute UI blocks backed by 5-minute recording segments.
- Events are stored as timestamps and can jump playback to the correct segment.
- The app receives only short-lived playback URLs, never OSS credentials.
- Empty, loading, and error states are handled cleanly.
- Existing live view, snapshot, PTZ, protection, and activity features continue to work.
