# Media Storage Flow

SentinelEdge stores media metadata in the database and stores video bytes outside the database.

## Storage Types

| Type | Meaning |
|---|---|
| `local_edge` | Media exists on the laptop edge service or local device storage. |
| `oss` | Media was uploaded to Alibaba OSS. |
| `pending_upload` | Metadata exists, but upload is not complete. |

## Clips vs Recordings

| Media | Typical storage | Purpose |
|---|---|---|
| Event clips | OSS when escalation/review is needed | Short reviewable media tied to an event. |
| Daily recordings | Local edge by default | Continuous or scheduled local history. |
| Thumbnails/keyframes | OSS or local edge | Quick preview for event list/detail views. |

## Direct-to-OSS Clip Flow

1. Edge detects an event.
2. Edge submits event metadata.
3. Edge asks backend for signed upload URL.
4. Backend creates `clips` row with `pending_upload`.
5. Backend returns short-lived signed upload URL.
6. Edge uploads directly to OSS.
7. Edge calls upload complete endpoint.
8. Backend validates metadata and marks clip `available`.

Relevant fields:

| Field | Use |
|---|---|
| `storage_type` | `pending_upload` before completion, `oss` after upload. |
| `oss_object_key` | Permanent OSS object key. |
| `upload_id` | Upload session identifier. |
| `upload_started_at` | Upload session start time. |
| `upload_completed_at` | Upload completion time. |
| `upload_expires_at` | Signed upload URL expiration. |
| `upload_error` | Last upload failure detail. |
| `status` | `pending_upload`, `uploading`, `available`, `failed`, or `deleted`. |

## Playback Flow

Clients should not store permanent OSS URLs.

1. User opens event detail.
2. Web app requests clip metadata.
3. Web app calls `POST /api/v1/clips/{clip_id}/signed-url`.
4. Backend validates user ownership and clip status.
5. Backend returns short-lived playback URL.

Signed playback URLs should expire quickly. The current default is 900 seconds.

When Alibaba OSS settings are configured, the backend signs OSS upload,
playback, and download URLs directly. Local/offline tests intentionally return
`placeholder://` URLs so the media flow can run without cloud credentials.

## Local Recording Flow

Daily or continuous recordings remain local unless selected for upload.

The edge service registers metadata:

```json
{
  "storage_type": "local_edge",
  "storage_path": "recordings/dev_frontdoor_001/2026-06-11/rec_001.mp4",
  "status": "local_only"
}
```

The backend can show that a recording exists without having the bytes. Playback from local edge requires a later relay, upload, or LAN access design.

## Retention

Enforced defaults:

| Media type | Retention | Stamped as |
|---|---|---|
| Event clips (incl. thumbnails) | `MEDIA_RETENTION_DAYS` (7 days) | `clips.expires_at` at upload-URL creation |
| Daily local recordings | `DAILY_RECORDING_RETENTION_HOURS` (72 hours) | `recordings.retention_until` at registration |

Enforcement has three layers:

1. **Read-time check** — signed-url/download endpoints return `410` (`clip_expired` / `recording_expired`) once the stamp passes, before any sweep runs.
2. **DB sweep** — `media_retention_service.run_sweep_loop` (started from the app lifespan, cadence `MEDIA_SWEEP_INTERVAL_SECONDS`, `<= 0` disables) soft-deletes expired rows (`status="deleted"` + `deleted_at`) and fails `pending_upload`/`uploading` clips older than 24 hours. It marks rows only and never talks to OSS.
3. **OSS lifecycle rules** — `scripts/deployment/media-bucket.ps1` applies bucket rules that delete `events/` objects after `MEDIA_RETENTION_DAYS` and `recordings/` objects after 3 days. Rerun it whenever `MEDIA_RETENTION_DAYS` changes (it reads the repo `.env`; pass `-EventRetentionDays` when KMS is authoritative).

Drift note: OSS lifecycle counts from object last-modified time and evaluates roughly daily, so bytes can outlive `expires_at` by a day or two. That is the safe direction — the DB and read-time checks always expire first, so nothing playable ever points at deleted bytes. The reverse (bytes gone, row still live) cannot happen. Worst case, a signed URL minted just before expiry stays fetchable for up to `SIGNED_URL_TTL_SECONDS` (900 s).

Deletion rules:

- Set `deleted_at` for soft delete.
- Block playback when `deleted_at` is set.
- Keep audit records for user-triggered or system-triggered deletion.

## Object Key Pattern

Recommended OSS keys:

```text
events/{user_id}/{device_id}/{event_id}/clip_{clip_id}.mp4
events/{user_id}/{device_id}/{event_id}/thumb_{clip_id}.jpg
recordings/{user_id}/{device_id}/{yyyymmddThhmmssZ}_{recording_id}.mp4
```

Object keys should be generated by the backend, not trusted from clients.

## Integrity Checks

When available, store:

- `file_size_bytes`
- `mime_type`
- `checksum_sha256`

For upload completion, compare the reported values against the metadata registered before upload.
